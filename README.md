# 基于开源项目的继承开发与开源合规说明

## 1. 文档目的

本文档用于说明本团队在课程实践中，如何基于开源项目进行继承开发、功能扩展与工程维护，并阐明在原仓库维护者不开放上传权限或不接受合并请求的情况下，团队如何通过自建项目仓库开展实践，同时不违背开源精神和课程要求。

## 2. 课程实践背景

课程实践强调利用开源社区资源完成软件系统开发，要求团队对开源软件进行检索、阅读、分析、维护和扩展，并完成需求、设计、实现、测试与文档编写等完整的软件工程活动。课程指导书明确提出：团队需要泛读 10000+ 行代码、每人精读和标注 1000+ 行代码、团队维护和新增 1000+ 行代码，并借助 GitHub、码云等平台建立自己的项目，完成代码提交、合并和工作量统计。[file:4]

课程教学材料同时强调“重用开源软件”“学生创建项目”“建立项目版本库”，并将重点放在需求构思、建模设计、代码实现、测试验证和成果文档质量上，而不是要求必须向原项目主仓库成功提交并被合并代码。[file:2]

## 3. 原始开源项目概述

本团队选取的基础项目为“基于 SSM 的校园二手交易平台”。该项目采用 Java 1.8、MySQL、Spring、Spring MVC、MyBatis、Tomcat、Thymeleaf、Maven 和 GitHub 等技术栈，面向校园场景实现二手商品交易、求购信息发布、商品检索、购物车、用户登录注册和个人中心等功能。[file:38]

从 README 内容可见，该项目已经实现了较完整的业务模块，包括商品首页、二手商城、商品详情、站内搜索、发布商品、个人信息、我的发布、购物车以及登录注册等页面与功能，具备较好的可读性和二次开发基础。[file:38]

## 4. 对老师要求的理解

结合课程指导书与课程实践材料，可以将老师的要求理解为：团队应当在开源软件基础上开展“维护性开发”和“新增功能开发”，而不是简单复现已有项目，更不是完全脱离开源基础另起炉灶。课程要求的核心在于：

- 能够围绕明确的软件需求，重用并分析现有开源项目；
- 能够在原有系统上进行结构理解、缺陷修正和功能扩展；
- 能够按照软件工程方法输出需求、设计、实现、测试和文档成果；
- 能够借助代码托管平台建立自己的团队项目仓库，开展协同开发。[file:4][file:2]

因此，课程实践的重点是“在开源基础上做实质性工程工作”，而不是“必须拿到原作者仓库的写权限”。

## 5. 如果原作者不开放上传权限，应如何处理

如果原 GitHub 仓库作者不提供协作者权限，或者不接受团队的代码合并请求，本团队仍然可以在遵守开源许可证和保留原项目来源信息的前提下，建立自己的项目仓库并继续维护开发。这样的做法与课程指导书中“借助 GitHub、码云等建立自己的项目，完成代码提交、合并操作”的要求是一致的。[file:4]

也就是说，原项目作者是否允许直接向其主分支上传代码，并不影响本团队开展课程实践。团队完全可以采用“基于原项目进行二次开发”的方式，在自己的仓库中持续演化系统，并在报告中清晰说明继承来源、修改内容和新增功能范围。[file:4][file:2]

## 6. 如何做到不违背开源精神

为了在课程实践中既满足教学要求，又不违背开源精神，团队应遵循以下原则：

### 6.1 明确标注项目来源

在团队项目 README、课程报告和答辩材料中，明确写出原始开源项目名称、原仓库地址、原作者信息，以及本团队与原项目之间的关系，例如“本项目基于某某开源项目进行继承开发和功能扩展”。

### 6.2 保留原有许可证与版权声明

如果原项目附带 LICENSE 文件、版权声明或相关 notice，团队应保留这些内容，不应删除、覆盖或故意弱化原作者贡献。若新增模块由团队独立实现，可以在不冲突的前提下补充团队自己的说明，但不能把原项目整体描述为“完全原创”。

### 6.3 区分原功能与新增功能

在说明文档中，需要清楚区分以下三类内容：

- 原开源项目已经实现的功能；
- 团队对原功能做出的缺陷修正或结构优化；
- 团队新增实现的功能模块。

这种区分有助于体现团队真实工作量，也符合课程对“维护和新增 1000+ 代码”的要求。[file:4]

### 6.4 遵守原有代码风格与工程结构

课程指导书明确要求新增代码必须遵循原有代码风格并保证质量，同时建议尽可能通过独立模块封装新增功能、保证高内聚与功能独立性，并考虑与原系统的良好集成。[file:4]

因此，团队在二次开发时，应尽量沿用原项目已有的包结构、命名规范、接口组织和编码风格，避免写出与原工程严重割裂的新代码。

### 6.5 尽可能回馈社区

如果团队对原项目进行了有价值的缺陷修复、文档补充或功能增强，可以选择向原项目提交 Issue、Pull Request 或修正建议。课程指导书也明确提到：如果向原开发团队提交修正建议或代码并被采纳，将获得较高加分。[file:4]

需要注意的是，“提交贡献”是积极行为，但“是否被采纳”并不是团队继续维护自己版本库的前提条件。

## 7. 本团队建议采用的实施方案

结合课程要求与当前项目情况，建议采用如下实施方案：

1. 以原始开源项目为基础完成部署、阅读和功能梳理；
2. 在团队 GitHub 账号或组织下创建新的课程实践仓库；
3. 在 README 中写明“本项目基于 Used-Trading-Platform 开源项目进行继承开发”；
4. 保留原项目必要的许可证、版权与来源说明；
5. 在原功能基础上增加课程要求的新功能模块；
6. 对修改点、修复点和新增点进行提交记录与工作量统计；
7. 输出《软件需求规格说明书》《软件设计规格说明书》《团队自评报告》《个人自评报告》和实践汇报材料。[file:4][file:38]

## 8. 结合本项目的具体写法

针对“校园二手交易平台”这一开源项目，团队可以在原有商品展示、搜索、发布、购物车和用户管理功能基础上，扩展更符合课程实践要求的新模块，例如：

- 商品审核与违规信息管理；
- 敏感词与敏感图检测增强；
- 求购信息智能匹配；
- 商品推荐与个性化推送；
- 交易状态追踪与确认；
- 管理员后台与统计分析模块。

这样既保持了与原项目的继承关系，又体现了团队对原系统的真实维护和功能扩展。[file:38][file:4]

## 9. 可直接写入报告的说明文字

本项目基于 GitHub 开源项目“Used-Trading-Platform”进行继承开发。原项目实现了校园二手交易平台的基础功能，包括商品展示、商品详情、站内搜索、发布商品、发布求购、购物车、个人中心和登录注册等模块。[file:38]

根据课程实践“重用开源软件、维护并新增代码、建立团队项目版本库、完成协同开发与文档输出”的要求，团队在充分阅读和分析原项目的基础上，自建课程实践仓库，对系统进行维护性修改与新增功能开发。[file:4][file:2]

若原项目维护者不提供写权限，或未接受团队提交的合并请求，团队仍将在遵守原项目开源许可证、保留原作者信息和明确标注继承关系的前提下，于自身仓库持续完成系统演化。这种方式符合开源协作常规，也不违背开源精神。[file:4][file:2]

## 10. 结论

综上，课程实践的本质要求是在开源基础上开展真实的软件工程活动，而不是要求团队必须成为原仓库协作者。只要团队能够明确来源、遵守许可证、保留原作者信息、清晰区分继承内容与新增内容，并在自己的仓库中完成规范的维护和扩展开发，就既符合课程要求，也符合开源精神。[file:4][file:2][file:38]
